# MemoWeft 治"慢/勤" + 落实三原则 · 给 Coder

> **来源**：dogfood 诊断 + 所有者三条原则 +「攒批更新」决定（2026-07-01）。
> **重点已重排**：核心不是"让更新变快"，而是"**别太勤 + 别挡人 + 别脑补**"。
> **本次授权（所有者最新指令）**：碰核心的点**自己定合理默认、直接做**，把选择在代码注释 / LOG 里标清楚即可，**不用先回来问**。

## 1. 三条原则（设计依据）

- **一 · 事件立即写、画像慢慢算**：事件/画像/归因三层不混在一个同步请求。
- **二 · 画像是缓存、别让用户等**：证据库是本体，画像是其上的认知缓存，可延迟/异步/重试。
- **三 · 归因别太勤**：越勤越胡说；应"记录 → 等更多证据 → 反复出现再归因"。谨慎观察者，非算命先生。

## 2. 现状对照（偏差 3 处）

- ✅ 读写已解耦、画像是可重建缓存、趋势"够频才归纳"——原则大半已贯彻。
- 🟡 **偏差①（核心）· 更新太勤**：`scheduleBackgroundUpdate`（`server.mjs:106`，`BG_DEBOUNCE_MS=7000`）——**每次聊完停手 7 秒就更新一次画像**，跑 3~4 趟模型。这不是"攒批"，是"一聊完就算"，又慢又费。
- 🟡 **偏差② · 手动按钮同步等**：`/api/refresh`（`server.mjs:222`）跑完才返回、逼用户干等。
- 🟡 **偏差③ · 归因每次都推**：`attribute` 每次 updateProfile 都归因、现象一出现就推（只标低置信）。

## 3. 做什么（按优先级）

### 🔺 核心 1 · 更新时机改「攒批」（治"勤"，落实原则一/二）
- 把"停手 7 秒就更新"改成 **攒够一批 + 歇久了、先到先触发**：
  - **攒够 N 条**新事件（未消化）→ 立即排更新；**或** 用户**空闲够久**（如 30 分钟没新消息）→ 更新一次。两者先到先触发。
  - `N`（批量阈值）和空闲时长**配置化**（`config.ts`，替代/扩展现 `BG_DEBOUNCE_MS`）。
- 效果：一天/一批才更新一次，慢几分钟也无所谓（后台、没人等）。

### 🔺 核心 2 · 归因加「攒够才推」门槛（治"脑补"，落实原则三）
- `attribute` 现状：每次都归因、一出现就推。改成——现象要 **反复出现 / 攒够证据**才归因（借趋势的"窗口内 ≥N 次"门槛，N 配置化）。没到门槛就"**只记录、等更多证据**"。
- ⚠️ 改归因行为，**dogfood 必须重验**：门槛别设太高变"永不归因"；验"该归的还归得到、不该急的不急"。
- 和现有"低置信假设 + 衰减/过期"别重复兜底——你自己权衡协调。

### ▪ 顺手（小成本，一起做）
- **3 · 手动"更新画像"改不阻塞**：`/api/refresh` 触发后台即刻返回，前端靠 `/api/bg-status` 轮询。别让用户干等。（攒批后手动按钮已次要，但改成不阻塞很轻。）
- **4 · 补落盘**：更新画像记 `distill/consolidate/attribute` **各步耗时(ms) + 结果**（扩 `runLog.ts`）。以后能读日志诊断。

### 🔻 降级 · 可选/往后
- **5 · 写路径配小快模型**：攒批后"慢"已不要命，**此条降优先级**——有空再做，或留作"可切换模型"架构的独立一步（写路径独立配模型、为档2 按授权路由留口，别写死俩固定 client）。

## 4. 不做

## 6. 验收标准

- [ ] **不再每次聊完就更新**——攒够 N 条 或 空闲够久才触发（核心 1）。
- [ ] 归因**不再"一句好累就推"**——攒够/反复才出假设；dogfood 验"变谨慎、又没漏该归的"（核心 2）。
- [ ] 手动"更新画像"立即返回、不卡界面；更新画像各步耗时落日志。
- [ ] `typecheck` / `test` / `build` 全绿。
- [ ] （若做了小模型）后台更新更快、对话质量不受影响。
