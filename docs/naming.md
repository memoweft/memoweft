# 命名与定位 · Naming & Positioning

> 本文是 MemoWeft 对外/对内用词的**唯一参照**。写 README、测试台 UI、宣传文案前先读这里。
> 一句话原则：**换词是为了好懂，但不能为了好懂牺牲可信。** 代码冷静、克制、工程化；温柔、拟人、好懂留给宿主（如星瑶）那一层。

---

## 1. 为什么要「双层命名」

MemoWeft 是**被 `import` 的库**，不是 App。它的读者天然分两拨，用词不能一刀切：

- **开发者**：要知道 `import` 什么、数据结构是什么、`evidence / event / cognition` 各对应哪张表、`updateProfile` 干了什么。给他们看「她记得你」只会让他们懵——**工程词必须保留**。
- **普通用户**：不关心表结构，只想知道「它到底为我做了什么」。给他们看 `cognition / attribution / confidence` 只会觉得「这是什么后台系统」——**这一层要翻译成用户能感受到的话**。

所以是**两层名字、指同一件事**，靠下面的词表互相映射。不是把架构名改软，也不是把用户文案写硬。

---

## 2. 定位话术（别吹大）

MemoWeft 的核心纪律是 **记 ≠ 信**：记下来不等于采信，推测只能是低把握度候选。所以对外话术要**克制**，别承诺「真正理解你」。

- ✅ 推荐（中文）：**用户理解层** —— 帮 AI 助手记住用户说过的话，并分清哪些能信、哪些只是猜、哪些要再确认。
- ✅ 推荐（英文）：**user-cognition layer** / user context & cognition layer。
- ✅ 稳的说法：「它让 AI 不只是『记住』，而是知道哪些可以相信、哪些只是猜测、哪些需要再问你确认。」
- ❌ 别写：「MemoWeft 让 AI **真正理解**你。」「它**懂**你。」——过度承诺，开发者会本能警惕、用户会失望。
- ⚠️ 「记忆 / memory」可以用，但**不能只用**——只叫「记忆」会被当成又一个 vector store。MemoWeft 值钱的是**证据与判断分开、每条判断可溯源、冲突不自动覆盖、把握度自算、临时状态会衰减**，这是「理解层」而非「记忆库」。
- 🧵 织布 / 经线纬线（warp / weft）隐喻只在 README 开头点**一句**，别到处用——普通用户对隐喻没感觉，他们要的是下面五句大白话。

**用户真正听得懂的五句（对外文案基准）：**
1. 它记得你说过什么。
2. 它知道哪些只是它暂时的猜测。
3. 它发现前后矛盾会来问你。
4. 它不会把过期的状态一直当成现在的你。
5. 它能让不同 AI 角色都继承同一份对你的了解。

## 3. 词表（对内工程词 ↔ 对外用户词）

同一个概念，三列各取所需。**源码 / API / 类型名只用第 1 列**；开发者文档用第 1、2 列并列；用户界面用第 3 列。

| 工程词（代码 / API，不改） | 开发者文档（中文并列） | 用户界面（对外产品词） |
| --- | --- | --- |
| `evidence` | 证据 | 记忆线索 / 原话 / 记录 |
| `event` | 事件 | 经历片段 / 一次经历 |
| `cognition` | 认知条目 | 对你的理解 / 理解条目（UI 里也可叫「理解卡片」） |
| `profile` | 用户画像 | 对你的了解 / 个人上下文（**用户界面避免「用户画像」**） |
| `confidence` | 置信度（0–1000） | 把握度（用户侧用定性档，见 §5） |
| `credStatus` | 可信状态 | 候选 / 低把握 / 有一定把握 / 比较确定 / 有冲突需确认 |
| `recall` | 召回 | 想起相关内容 / 相关记忆 |
| `attribution` | 归因 | 猜测原因 / 可能是因为 |
| `conflict` | 冲突 | 需要确认的矛盾 / 说法不一致 |
| `hypothesis` | 假设 | 暂时的猜测 / 待确认的想法 |
| `decay / expire` | 衰减 / 过期 | 变淡 / 不再当成现在的你 |
| `updateProfile` | 更新画像（distill→consolidate→attribute→index） | 整理记忆 / 重新认识你 |
| `distill` | 事件化 | 整理成经历片段 |
| `consolidate` | 增量消化 | 更新理解 |
| `retriever` | 召回底座 | （不建议给用户看；对内叫「找相关记忆」） |
| `asking / proposeAsk` | 主动询问 | 想问你的问题 |
| `revisitConflicts` | 复看冲突 | 检查矛盾 |
| `aggregateTrends` | 趋势聚合 | 总结最近状态 |
| source trace | 溯源 | 根据哪句话知道的 / 从哪里看出来的 |

> ⚠️ `cognition` → 「理解卡片」只适合**纯 UI**；在 API / 开发者文档里别叫「卡片」，免得被当成前端组件——用 `Cognition` / 认知条目 / 理解条目。

## 4. 各场景用哪层词

| 场景 | 用词策略 |
| --- | --- |
| 源码 / API / 类型名 | **只用工程词**：`evidence / event / cognition / confidence / retriever …` |
| 开发者文档（architecture / integration / INSTALL） | 中文解释 + 英文原词并列，保留精确术语 |
| README 首页 | 少量用户语言开场 + **明确说清它是库**；配一张本词表的对照 |
| 测试台 · 用户模式 | 用户词：原话 / 经历 / 对你的了解 / 想起 / 把握度（定性档） |
| 测试台 · 开发者模式 | 显示原始字段名与数值：`confidence: 640 / credStatus: limited`，软词后括注工程词 |

## 5. 把握度（confidence）怎么说

代码里 `confidence` 是 **0–1000**，给算法调参用；`credStatus` 分档。但 **0–1000 对普通人没有感觉**。

- **用户模式**：别显示原始数字。用定性档：**低把握 / 有一定把握 / 比较确定 / 已较稳定 / 有冲突，需确认**。（当前测试台用户模式已直接隐藏所有内部分数，只显示人话——这就是对的。）
- **开发者模式**：显示 `confidence: 640` + `credStatus: limited`，方便调参。

按钮/进度也别暗示「点一下立刻变聪明」。写路径要调几次模型、**是慢活**，UI 要诚实：「正在整理记忆，可能需要一会儿。整理完之后，它在后续对话里会更容易想起相关内容。」

## 6. MemoWeft ↔ 星瑶：边界别打乱

这是最容易搞混、也最该守住的一条：

- **MemoWeft = 能力层**：冷静、克制、工程化。它**不是「她」**，不聊天、不做人设、不主动关心。文案主语是「宿主可以用 MemoWeft 提供的上下文，让角色更连续、更不失忆」。
- **星瑶 = 人格 / 体验层**：温柔、拟人、好懂。第一人称「我记得你之前说过……」是**星瑶**说的，不是 MemoWeft 说的。

所以：**MemoWeft 的任何文案都不出现「她」**。拟人和情感放到宿主那一层。MemoWeft 最值钱的地方，正是给星瑶垫一套**不乱信、不乱忘、不乱脑补**的底座。

---

## 7. 落地范围（本阶段）

先改**用户看得到的**：README 首页、测试台 UI、用户模式文案、文档标题。**代码里的核心术语先不动**（`evidence / event / cognition / updateProfile …` 原样保留）。等产品词在真实使用里稳定了，再决定要不要给 API 加别名——那是后面的事，不在本阶段。
