# MemoWeft 后续架构归位与实施路线

## 0. 文档目的

本文档用于统一 MemoWeft 后续开发方向。

当前 MemoWeft 已经完成了大量核心能力，包括 evidence、event、cognition、updateProfile、Conversation、Retriever、记忆管理页面、网页端测试台、感知采集等。

但目前存在一个核心问题：

> 很多能力已经做出来了，但层级边界不够清楚，Core、Host、Plugin 的职责混在了一起。

后续开发的重点不是继续堆功能，而是把已有能力重新归位，让 MemoWeft 从“开发现场”整理成一个清晰、稳定、可被宿主接入的记忆/认知框架。

---

# 1. 最终架构定位

MemoWeft 后续应拆成三层：

```txt
MemoWeft Core：记忆/认知内核
Host App：应用壳、权限、配置、网页端、记忆管理、插件管理
Plugins：角色、工具、感知采集等扩展能力
```

一句话定义：

> MemoWeft Core 负责记忆怎么正确存在，Host 负责用户怎么使用和管理，Plugin 负责扩展能力。

---

# 2. 三层职责边界

## 2.1 MemoWeft Core

MemoWeft Core 是记忆引擎，不是完整应用。

Core 应该负责：

```txt
evidence
event
cognition
updateProfile
Conversation
Retriever
LLMClient 抽象
ingestObservations
召回
置信度
纠正
冲突
失效
导入导出 bundle
图谱 payload
受控记忆管理 API
schema migration
integrity check
```

Core 不应该负责：

```txt
网页 UI
真实窗口采集
星瑶人格
插件系统
用户配置页面
记忆管理页面
完整宿主应用
插件市场
```

Core 的核心原则：

> Core 不长眼睛，也不长脸。它只负责接收标准输入，沉淀记忆，提供召回、更新和受控管理能力。

---

## 2.2 Host App

Host 是 MemoWeft 的应用主体 / 运行壳。

Host 应该负责：

```txt
网页端
聊天入口
配置向导
模型配置
数据目录
权限管理
记忆管理页面
插件管理
备份迁移页面
多会话
后台画像更新状态
调试面板
调用 MemoWeft Core
```

当前 testbench / 网页端 / 记忆管理页，其实已经是 Host 的雏形。

后续应该逐步迁移到：

```txt
apps/memoweft-host/
```

Host 的核心原则：

> Host 负责调度、权限、用户体验和插件管理，但不能绕过 Core 的记忆规则直接修改底层数据。

---

## 2.3 Plugins

Plugin 是 Host 的扩展能力，不是 Core 的一部分。

插件分三类：

```txt
Experience Plugin：星瑶、普通助手、工作助理、宠物助手
Tool Plugin：GitHub、文件、日历、浏览器、Shell
Collector Plugin：窗口采集、睡眠、心率、手机使用记录
```

插件可以做：

```txt
提交观察
请求读取认知
申请工具权限
建议某条记忆过期
请求用户确认一个假设
提供角色 prompt
提供工具能力
提供采集能力
```

插件不能做：

```txt
直接删除 evidence
直接修改 cognition
直接修改 allowCloudRead
直接把 cognition 标记为 stable
直接绕过 Host 写入敏感观察
直接管理记忆库
```

插件的核心原则：

> 插件只能请求，Host 审核，Core 执行记忆规则。

---

# 3. Core / Host / Plugin 的交互方式

MemoWeft Core 不应该写死 Host，也不应该知道具体有哪些插件。

Core 只认识标准输入输出：

```txt
UserMessage
Observation
MemoryCommand
MemoryQuery
RecallInput
UpdateProfileInput
```

Host 负责加载插件、判断权限、调用 Core。

标准数据流：

```txt
Plugin → Host → Core
Core → Host → Plugin / UI / LLM
```

例如感知采集流程：

```txt
Collector Plugin 采集窗口数据
  ↓
Host 检查权限和用户设置
  ↓
Host 调用 core.ingestObservation()
  ↓
Core 转换为 evidence
  ↓
后续 updateProfile / recall 可使用该 evidence
```

例如记忆管理流程：

```txt
用户在 Host 记忆管理页面点击“标记失效”
  ↓
Host 做二次确认
  ↓
Host 调用 core.memory.invalidateCognition()
  ↓
Core 标记 invalidAt、记录原因、维护索引
  ↓
后续 recall 自动跳过该 cognition
```

# 4. 当前已有功能的归位

## 4.1 感知功能

最终归位：

```txt
采集实现：Plugin
权限审核：Host
摄入口：Core
```

Core 可以保留：

```txt
Observation 类型
ingestObservations()
observed evidence 规则
授权字段
```

Core 不应该继续承担：

```txt
真实窗口采集
Win32 sampler
collector runner
PowerShell / 系统调用
定时采样循环
```

这些未来应迁移到：

```txt
plugins/collector-active-window/
```

或先标记为：

```txt
experimental collector implementation
```

---

## 4.2 记忆管理

最终归位：

```txt
底层规则 / API：Core
页面 / 操作流程：Host
插件：无直接管理权
```

Core 应保留：

```txt
invalidateCognition()
updateEvidenceAuthorization()
removeEvidenceSafely()
removeCognitionSafely()
mergeCognition()
archiveCognition()
checkIntegrity()
exportBundle()
importBundle()
buildMemoryGraph()
```

Host 应负责：

```txt
记忆列表
搜索筛选
详情弹窗
授权开关
删除确认
图谱 UI
备份迁移页面
用户确认流程
```

不要让 Host 直接写 SQL 或直接操作底层 store 完成关键管理行为。

错误方式：

```ts
cognitionStore.update(id, { invalidAt: ... })
```

推荐方式：

```ts
await core.memory.invalidateCognition({
  cognitionId,
  reason: 'user_rejected'
});
```

---

## 4.3 网页端 / testbench

当前 testbench 已经包含：

```txt
聊天入口
配置向导
记忆管理
备份迁移
多会话
调试区
后台画像更新
```

这些已经不只是测试工具，而是 Host 雏形。

后续应迁移到：

```txt
apps/memoweft-host/
```

testbench 可以保留为开发模式，但不应继续代表 Core。

---

## 4.4 星瑶

星瑶不属于 Core，也不应该写死在 Host 中。

星瑶应作为：

```txt
Experience Plugin
```

未来路径可以是：

```txt
plugins/experience-xingyao/
```

同时需要保留一个普通助手插件：

```txt
plugins/experience-plain/
```

这样可以证明 MemoWeft 不是星瑶专用记忆库，而是通用用户认知框架。

# 5. Core 后续优化任务

当前 Core 的优化目标不是继续变聪明，而是变得：

```txt
接口稳定
边界干净
管理受控
数据可迁移
召回不乱
隐私不漏
长期不膨胀
```

---

## 5.1 增加统一 Core Facade

需要新增统一入口，例如：

```ts
createMemoWeftCore()
```

或：

```ts
MemoWeftCore
```

Host 后续应优先通过这个入口调用 Core。

建议接口形态：

```ts
interface MemoWeftCore {
  ingestUserMessage(input: UserMessageInput): Promise<Evidence>;

  ingestObservation(input: ObservationInput): Promise<Evidence[]>;

  recall(input: RecallInput): Promise<RecalledCognition[]>;

  handleConversationTurn(input: ConversationInput): Promise<TurnOutcome>;

  updateProfile(input: UpdateProfileInput): Promise<UpdateProfileResult>;

  memory: MemoryManagementAPI;

  portable: PortableMemoryAPI;

  graph: MemoryGraphAPI;
}
```

目标：

> Host 不应该到处直接碰 SqliteEvidenceStore、SqliteCognitionStore 等底层实现。

---

## 5.2 收瘦 Core 主入口导出

Core 主入口应该导出：

```txt
createMemoWeftCore
stores
Conversation
updateProfile
Retriever
LLMClient
ingestObservations
exportBundle / importBundle
buildMemoryGraph
受控 memory API
```

不应该从主入口导出：

```txt
真实窗口采集器
Win32 sampler
collector runner
网页端 UI
Host 页面
```

真实 collector 可以先标记为 experimental，或者后续迁移到独立插件包。

---

## 5.3 建立受控记忆管理 API

记忆管理不是普通 CRUD。

Core 需要提供正式 API：

```txt
invalidateCognition()
updateEvidenceAuthorization()
removeEvidenceSafely()
removeCognitionSafely()
mergeCognition()
archiveCognition()
checkIntegrity()
exportBundle()
importBundle()
```

这些 API 需要负责：

```txt
维护引用完整性
记录操作原因
处理索引更新
保证 recall 不注入 invalid cognition
保证授权变更后 prompt 过滤正确
处理 evidence 删除后的关联影响
```

---

## 5.4 补 schema migration / integrity check

这是长期使用必须补的硬债。

需要实现：

```txt
schemaVersion
migration runner
迁移前备份
migration log
dry-run
integrity check
orphan evidence / cognition 检查
bundle schema compatibility
```

注意：

```txt
vector index 是派生资产，可以重建
evidence / event / cognition 是用户资产，不能粗暴删除重建
```

---

## 5.5 召回质量 v2

召回优化方向：

```txt
按 purpose 召回
按 scope 召回
按 contentType 过滤
相似度阈值
召回理由 explain
用户负反馈
关键词 + 向量 + 时间 + 类型混合检索
```

召回目标：

> 不只是能记住，而是不能乱用记忆。

需要避免：

```txt
无关记忆被注入
过期状态被注入
低置信 cognition 被当成事实
跨 subject 串线
敏感记忆被错误上云
```

---

## 5.6 写路径膨胀治理

长期使用后 updateProfile 可能越来越慢、prompt 越来越长。

需要治理：

```txt
consolidate 不要每次喂全部 active cognition
只喂与当前事件相关的 cognition
相似 cognition 合并
长期画像分层
低价值 cognition 降权 / 归档
promptChars 阈值报警
profileSize 控制
```

目标：

> 记忆不能越用越肥，画像不能变成垃圾堆。

---

## 5.7 cloud / local 权限分流

未来 Host 可能同时使用云端模型和本地模型。

Core 需要支持：

```ts
type ModelTier = 'cloud' | 'local';
```

规则：

```txt
cloud 模型不能读取 allowCloudRead=false 的 evidence
local 模型可以读取 allowLocalRead=true 的 evidence
是否能用于画像推理还要看 allowInference
```

所有 prompt 构建前都应该根据模型 tier 过滤记忆。

原则：

> 不是所有模型都有同样的记忆读取权。

## 6.3 Host 调 Core，不直接改库

Host 的所有记忆操作必须通过 Core API。

正确方式：

```ts
await core.memory.invalidateCognition(...);
await core.memory.updateEvidenceAuthorization(...);
await core.ingestObservation(...);
await core.recall(...);
```

错误方式：

```ts
db.prepare("UPDATE cognition SET ...").run(...)
cognitionStore.update(...)
evidenceStore.remove(...)
```

除非是 Core 内部实现，否则 Host 不应直接操作底层数据。

# 7. Plugin 后续任务

## 7.1 定义最小插件契约

Host 稳定后，再定义插件契约。

第一版不要做复杂插件市场，只做最小可用契约。

建议：

```ts
interface MemoWeftPlugin {
  id: string;
  name: string;
  type: 'experience' | 'tool' | 'collector';

  permissions?: PluginPermissions;

  systemPrompt?: string;

  onLoad?: Function;
  onUserMessage?: Function;
  onObservation?: Function;
}
```

插件拿到的不是 Core 全权限对象，而是受限上下文：

```ts
interface PluginContext {
  submitObservation(input: PluginObservation): Promise<void>;

  requestMemory(query: MemoryQuery): Promise<RecalledCognition[]>;

  requestPermission(request: PermissionRequest): Promise<boolean>;

  emitUIEvent(event: PluginUIEvent): void;
}
```

---

## 7.2 默认插件

至少需要两个体验插件：

```txt
experience-plain：普通长期记忆助手
experience-xingyao：星瑶陪伴型助手
```

这样证明 MemoWeft 是通用框架，而不是星瑶专用。

后续再考虑：

```txt
collector-active-window
work-assistant
github-tool
file-tool
calendar-tool
browser-tool
```

但不要太早做一堆插件。

# 12. 最终定稿

MemoWeft 后续所有任务都围绕三者归位展开：

```txt
MemoWeft Core：记忆引擎
Host App：运行壳
Plugins：外接能力
```

一句话定稿：

> MemoWeft Core 是记忆引擎，Host 是运行壳，Plugin 是外接能力。Core 通过稳定接口提供记忆能力，Host 负责权限、调度和用户体验，Plugin 负责角色、工具和感知采集。
