## 0. 一句话

v0.6 把 MemoWeft **理解人机对话**的能力从 D-0033 的「临时 AI 上下文后缀」升级成**结构化的交互语义模型**:每条用户证据配一份**语义解析**(这句是肯定/否定/选择/含糊、在回应谁提出的什么命题),**由代码据此计算 `formedBy`**(不再让 LLM 报),从而既能听懂短回答("是"/"后者"/"可能吧"),又用**结构墙**挡死"AI 编 X、用户一句'是'洗成事实"的自我印证。

---

## 1. 背景与要解决的问题

当前链路 `Evidence → Distill → Event → Cognition` 存在两个对立缺口:

- **听不懂短回答**:`AI:你喜欢研究 AI 吗? 用户:是` → 系统只存"是",不知在回应啥 → 存不了真实自我披露。
- **不能让 AI 输出直接变事实**:`AI:你是不是聪明、有创造力、注定成功? 用户:是` → 绝不能形成"用户注定成功"。

D-0033 已经用 `preceding_ai_context`(上一轮 AI 那句作只读后缀)+ `confirmed` 来源强度部分解决了第一个缺口。但 v0.6 要把它**结构化、可审计、并在真实产品里真正生效**(见 §8 头号风险)。

目标链路:

```
Conversation
   ↓
Interaction Episode
   ↓
Evidence + Interaction Context
   ↓
Semantic Resolution
   ↓
Event
   ↓
Cognition
```

## 3. 核心设计(提案 · 待批)

### 3.1 三个新概念的数据落点

| 概念 | 落点 | 说明 |
|---|---|---|
| **Interaction Episode** | `interaction_context` 表的 `episode_id` 列 | 一段完整人机交互;**不产 Cognition**,只提供语义环境。边界归属见 §10.2(待你拍板) |
| **Interaction Context** | `interaction_context` 表 | 保存**用户可见**上下文(assistant 可见回复 / user message / 可见 tool 结果);**禁**收 system prompt / 隐藏提示 / CoT / 记忆注入 / 内部工具入参 |
| **Semantic Resolution** | `semantic_resolution` 表(1 证据 ↔ 1 解析) | 把 `Evidence + Context` 解析成「可理解语义」;**是解释结果,不是新的事实来源** |

两张表的列(照规范 §7):

```
interaction_context:   id, conversation_id, episode_id, context_json, context_hash, created_at
semantic_resolution:   id, evidence_id, resolved_content, response_act, prompt_act,
                       proposition_origin, assertion_strength, required_context,
                       resolver_version, created_at
```

### 3.2 代码派生 `formedBy` 的映射(v0.6 的认知内核 · 提案)

现状:`formedBy` 由 LLM 报 `formed_by` 字段、代码 `VALID_FORMED` 白名单校验后采纳(`consolidate.ts:89,96`);`ruled` 由 `trends.ts:129` 代码钉、`confirmed` 由 reinforce 升级路(`consolidate.ts:233-240`)代码算。

v0.6 用 `deriveFormedBy(supportEvidences)`(新函数)**取代 LLM 报**,输入 = 每条支撑证据的 `sourceKind`(证据层已有)+ 其 `semantic_resolution` 三维,规则如下:

| 证据来源 / 语义 | → formedBy | 底分 |
|---|---|---|
| `sourceKind=observed` | `observed` | 350 |
| `sourceKind=tool` | `observed`(**绝不 stated**) | 350 |
| `spoken` ∧ `proposition_origin=user_stated` ∧ `assertion_strength=explicit` | `stated` | 600 |
| `spoken` ∧ `proposition_origin=assistant_proposed` ∧ `response_act=affirm` ∧ `explicit` | `confirmed` | 280(封顶 480<limited) |
| `spoken` ∧ `assistant_proposed` ∧ `affirm` ∧ `weak`("可能吧") | `confirmed`(弱)或**不形成** | — |
| `response_act=negate`(否定/纠正) | 走 correct 路;新内容按 `user_stated` 定 | — |
| 复合命题(AI 一次抛多个) | **拆分**:只对明确点头的原子产 `confirmed`,含糊的不产 | — |
| 其它 / 无法解析 | `inferred`(保守兜底) | 200 |
| **趋势(频率规则聚出)** | `ruled` — **不动 `trends.ts`,保持代码路径** | 450 |

**安全不变式**(必须由结构保证、非提示词软判):`confirmed` 只能来自 `assistant_proposed + affirm`,底分 280 + 支持满 200 = 480 < limited 500 → **纯附和永远顶天"低置信"**;只有用户后来 `user_stated` 主动印证才经升级路破 480(沿用 D-0033,统一到 `deriveFormedBy`)。

### 3.3 resolver 融进 distill/consolidate(决策 ③)

不新建独立 LLM step。在 distill(或 consolidate)已有的 LLM 调用里**扩展输出 JSON**,让模型对每条证据附一份 `{response_act, proposition_origin, assertion_strength, resolved_content, required_context}`,代码落 `semantic_resolution` 表。`deriveFormedBy` 读它。既有 eval 语料无此维度 → 缺省保守(→ inferred/现行为),Phase 2 提示词教会后重跑。

**代码派生的精确 seam**(cognition 探针核实):consolidate `new` 分支 `199-201` 之间(`pickSupport` 求出支撑证据之后、`computeConfidence` 之前),`correct` 分支 `255-258` 之间。

## 5. API 影响(api-freeze)

`tests/api/api-surface.snapshot` 记**类型结构**(非值)。已核实的关键点:

- **枚举增值在快照里不透明**(`type FormedBy = FormedBy`)。若 v0.6 新增 `ResponseAct`/`PropositionOrigin`/`AssertionStrength` 等**联合类型别名并导出**,它们那一行同样不透明,**api-freeze 不会自动红** → 必须**人工同步契约**(别把 api-freeze 当护栏)。'confirmed' 当初触发红,唯一原因是它进了 `config.baseByFormedBy` 被结构展开。
- **以下改动都 additive(非破坏)但都改快照 → 都要走 api:update + 全套治理**:
  - `ConversationInput` 加 `episodeId?`(可选字段,快照 line 77 那行变)。
  - `UserMessageInput` 加 `conversationId?` / `episodeId?`(关联会话,让 core 复用 working memory 抓上一轮 AI + 切 episode;**precedingAiContext 不暴露给宿主传,由 core 内部管**,见 §8)。
  - 新门面方法 `core.recordAssistantReply({ conversationId, content })`:宿主把自己生成的 AI 回复报告给 core(只进 working memory 作后续上下文,**永不落证据**,3a)。additive、触快照。
  - 新增导出类型 `SemanticResolution` / `ResponseAct` / `PropositionOrigin` / `AssertionStrength` / `InteractionContext`(若决定导出;能内部化就不进 index.ts、不动快照)。
  - `MemoWeftCore` 若加新方法(如 `resolveTurn`)→ 接口增成员 + 契约方法计数(现"25 methods")同步。
- **破坏性红线**:把任何**已有可选字段改必填**才是 breaking;新增可选字段/枚举增值不是。v0.6 全部按 additive 设计。
- **subagent 无法向人类取批准**(章程 8.4.1)→ 所有 api:update / D-xxxx 必须**主控 Integrator** 做。

## 6. 数据库影响

### 6.1 两张新表 — 照 `management_log` 模板,不上 formal v2

- 各自 store 构造函数里 `CREATE TABLE IF NOT EXISTS`,**挂共享连接**(才能进 `transaction()`,否则像 vectors 那样孤立),**不进 `migrations.ts`**、`LATEST_SCHEMA_VERSION` 继续留 v1。
- 理由(D-0023/D-0033 先例):新建整张表在 fresh 库与从 0.1.0 fixture 迁移的库两条路径都会跑 → `migrations.test.ts:87` schema 签名收敛测试天然满足;上 formal v2 反而要处理直接构造老库覆盖不到 + 撞号。

### 6.2 结构墙(3a/3d)— 复用 D-0033 哲学

- **核心不变式**:`semantic_resolution.resolved_content`、`interaction_context.context_json` 里的 AI 可见文本是**解释/上下文、不是证据**。**永不进 `validEvidence` / `pickSupport` 白名单、永不给证据 id、永不注入为可溯源证据**——这是 3a/3d,与「是否进便携包」正交(进包是数据迁移,不等于内容变证据)。
- 注入 distill/consolidate 时只作 LLM 输入的上下文文本(沿用 `aiContextSuffix` 的「只作后缀、不铸带 id 条目」),LLM 拿它理解语义,但派生认知的 support 只能引真证据 id。
- `preceding_ai_context`(D-0033 挂 evidence 的隐藏列):v0.6 的 `interaction_context` 是它的结构化超集;Phase 1 定**并存还是迁移**(实现细节)。

### 6.3 formedBy 全面接管的专项影响(最脆弱写路径)

- 删 `CONSOLIDATE_PROMPT` 的 formed_by 指令(`prompts.ts:39/60`)→ **bump `consolidate` v3→v4** → `tests/prompts/registry.test.ts` 哈希闸门变红 → 须 `npm run prompts:update` + **重跑 `bench/eval-consolidation.mjs` 全 42 场景、commit 正文贴前后分**(§15.3/D-0009)。
- `pickCognition`(`consolidate.ts:92-98`)不再读 `c.formed_by`(或保留作纯兜底);`deriveFormedBy` 的输出天然合法,`VALID_FORMED` 白名单对代码路径退化成 no-op。
- **必须保绿**的固化纪律:emotion-cap 封顶、conflict 暴露不裁决、fact-vs-belief、no-over-inference、chitchat-negative;`confidence` 仍规则自算、resolver 只产分类不产置信(**铁律 3b**)。
- `trends.ts` 的 `ruled` 不动(它本就不听 LLM,`:129` 硬钉 + `:80` 已排除 confirmed 防诱导灌 ruled)。

## 8. 头号风险:weftmate 全程绕开 Conversation 路

**已亲自核实(HEAD + 工作树 grep 皆空)**:

- `weftmate/src/memoweft.ts` 只 re-export 8 符号,`handleConversationTurn` **不在其中**。
- weftmate 的 `record`/`recordChat`/指正路**都走裸 `core.ingestUserMessage`**(`server.ts:343,351,1508`),只带 `originId`(如 `weftmate-agent:${taskId}`,**task 粒度**),从不传 conversationId/seedTurns。
- → **D-0033 的 `preceding_ai_context`/`confirmed` 在 weftmate 从未生效过**;memoweft `DECISIONS.md:487`「weftmate 走 handleConversationTurn(server.ts:366 ✓)」**记载错误**,需订正。

**应对(决策 ① + 人类澄清:weftmate 那层只有前端按钮、后端未做,授权 core 多承担)**:让 **core 承担会话上下文管理**——

- `ingestUserMessage` 加 `conversationId?`:带上它,core 就为该会话维护一份 working memory,存证据时自动抓上一轮 AI 作上下文 + 关联 episode(复用 D-0033 已有的先存后答 + `WorkingMemory`,只是从 `Conversation` 的「回话」里解耦出来,让不走回话的宿主也能用)。
- 新增 `core.recordAssistantReply({ conversationId, content })`:weftmate 生成 AI 回复后调它,core 把这轮助手回复 push 进该会话 working memory(**只作后续上下文、永不落证据**,3a)。
- weftmate 接线量 = **两处调接口**(ingest 时带上已有的 convId;AI 回复后调 recordAssistantReply),**不碰 AI 上下文的存储/结构墙细节、不碰 episode 切分**——全在 core 内部。这正合「core 能做就 core 做、weftmate 直接调接口」。
- 结果:v0.6 在真实产品**真正生效**,且 AI 文本永远在 core 内部流转(比让宿主传 precedingAiContext 更安全)。

> 连带:改 `UserMessageInput` → weftmate 唯一接缝 `memoweft.ts` typecheck 是第一道验证;additive 可选字段不破坏 weftmate 现有调用。weftmate 有未提交 WIP,协同须只读、不碰。

### 10.1 两张新表进便携包(已定:进)

- 决策:**进包**——交互上下文/语义解析是「对用户的理解」的一部分,跟随用户跨机器迁移。
- 动 `portable/{model,exportBundle,importBundle,validateBundle}.ts` + 给两个新 store 加 `insert` + **bump `BUNDLE_SCHEMA_VERSION` 1→2** + 旧包(无这两段)导入兼容(`?? []`)。

- 决策:宿主可选传 `episodeId`;不传时 core 按 `conversationId + idle 间隔` 自动切。weftmate 已有 session id(`s-<ms>-<seq>`)可映射,但即使不传也能用。
