## [Unreleased]

### Fixed

- **Whole batches of evidence were silently discarded because the model truncated evidence ids (`DECISIONS.md` D-0036)** — during Phase 4 dogfood, **5 of 15 real-conversation events consolidated to nothing** (0 resolutions **and** 0 cognitions, perfectly covarying) while still being marked `consolidated=1`, permanently losing **47 user utterances** — including one where the user plainly stated their age and car. Root cause, pinned by 6/6 controlled replays with zero counterexamples: the consolidate prompt's **output example uses a 4-character placeholder id** (`"support_evidence_ids":["ev-1"]`) while `buildMessages` actually feeds **36-character UUIDs** — so the model **intermittently imitates the example's id shape**, echoing back the UUID's **first 8 characters** (`"25601f4c"`), occasionally as `"ev-25601f4c"`. Both whitelists matched **exactly** (`pickSupport` via `validEvidence.has`, resolution narrowing via `spokenEvidence.has`), so every short id missed, every cognition was dropped at `support.length === 0`, every resolution was skipped — and `markConsolidated` then **unconditionally** marked the events done. **The model was doing its job; the code was throwing the results away.** Because both channels share one id whitelist (`spokenEvidence ⊆ validEvidence`), resolutions and cognitions **died together** — the exact "0 and 0 covary" signature that made this so hard to see. The **same** root cause produces a second symptom: when the model writes long ids in `resolutions` but short ones in `new`, you get **all resolutions, zero new cognitions**. **Zero signal by construction**: the JSON is fully valid, so `jsonRepair` never warned, `finish_reason=stop`, `llmCalls=1`, `completion_tokens` 5200–8500 — only the writes were empty. Fix: an internal `resolveEvidenceId` normalises ids — **exact match first** (verbatim behaviour when the model gets it right), falling back to a **unique-prefix** resolution (stripping a copied `ev-` prefix), and persisting the **resolved real id** (a short id in the table would be dirty data). **The whitelist is not loosened one inch (iron rules 3a/3d)**: resolution can only ever land **inside** the whitelist and must be **unambiguous** — fabricated ids (not a prefix of any real id), ambiguous prefixes (matching several), and over-short prefixes (< 8) are all still dropped. **Prompts are untouched**, so this does not trigger the §15.3/D-0009 prompt-version + full-eval gate. **End-to-end on the live model** (same input × 6): whole-batch losses **5/6 → 0/6**; new cognitions **all-zero → 10, 10, 11, 9, 9, 8**; zero dirty ids — while the **model's behaviour is unchanged** (it still writes short ids: 122 short vs 13 long). Pinned by `tests/evidenceIdTruncation.test.ts` (2 cases that real ids are no longer killed, 4 that the guardrails still hold), complementing `confirmedLaundering.test.ts`'s 3a cases. See `DECISIONS.md` D-0036.

### Added

- **`@memoweft/adapter-openai-agents` gains the v0.6 conversation-context line (`recordAssistantReply`) (`DECISIONS.md` D-0040)** — the `run` wrapper now completes MemoWeft's 0.6 story for the OpenAI Agents SDK. Pass a stable `conversationId` per run (`options.memoweft.conversationId`) and, when the Core is 0.6, the user's turn is ingested with it (so Core captures the **previous** turn's AI reply into `preceding_ai_context`) and after the run the wrapper reports the turn's **final AI reply** via `recordAssistantReply` — **context only, never evidence** (iron rule 3a): it lets the *next* turn's short reply ("yes" / "the latter") be understood, and is never stored as a memory. New exported helpers `finalAssistantText(result)` (extracts the reply text from `RunResult.finalOutput` or the last `message_output_item`) and `recordFinalReply(core, result, conversationId)` (self-gating: capability probe + `conversationId` + non-empty text, swallows a throwing `recordAssistantReply`). On memoweft `0.5` the whole line is skipped via a runtime capability probe, so `memoweft`'s peer range widens to `^0.5 || ^0.6`. **Scoped honestly (`DECISIONS.md` D-0040)**: this thin `recordAssistantReply` line — not a full `MemoweftSession` — is the deliberate ③ of the adapter-scaling batch; the SDK's announced next-gen would reshape Session, so a full Session implementation (duplicate surface + drift risk) is deferred, while this delivers the same 0.6 goal with zero new dependency and the smallest rework surface. **Purely additive** — no `conversationId` / a 0.5 Core behaves exactly as before; Core's `src/index.ts` API surface is unchanged (`npm run api:check` still passes). 8 offline unit tests pin the extraction and gating. See `DECISIONS.md` D-0040.

Conversation threading uses `runtime.configurable.thread_id` as the `conversationId` by default (overridable). The `langchain` umbrella (which ships the v1 middleware API) is an **optional** peer — the retriever + callback path still needs only `@langchain/core`, preserving D-0028's minimal-peer property; `memoweft`'s peer range widens to `^0.5 || ^0.6`. **Privacy/degradation** identical to the retriever path (`provenance` never injected, recall bounded by `recallTimeoutMs`, writes retry once, never throws to the agent). All seams pinned from `langchain@1.5.3`'s type definitions. Passes the adapter-kit contract as a second driver `langchain-mw` (AD-1…AD-9; `ad5`/`ad9` N/A). **Purely additive** — the existing retriever/callback API is untouched and Core's `src/index.ts` API surface is unchanged (`npm run api:check` still passes). See `DECISIONS.md` D-0039.

- **New adapter — `@memoweft/adapter-mastra` for Mastra (`DECISIONS.md` D-0038)** — a Processor adapter that wires MemoWeft's long-term memory into a [Mastra](https://mastra.ai) agent. `createMemoWeftProcessor(core, opts)` returns one `Processor` you register in **both** `inputProcessors` and `outputProcessors`. **Recall** runs in `processInput` (before the model): it recalls memory for the user's turn and injects it into the **system channel** (returns `{ messages, systemMessages }`) — the user message is never touched, so nothing injected can leak back as "what the user said". **Writes** run in `processOutputResult` (after the model answers): the user's turn → `spoken` evidence, each tool **result** → `tool` evidence (only `payload.result`, never `payload.args` — iron rule 3a by construction), and the assistant reply → `recordAssistantReply` (context-only, never evidence). The user utterance is **captured pre-injection in `processInput` and persisted in `processOutputResult`** (handed over via the processor's `state`, with a fallback to the output-phase messages) — this ordering is required by 0.6's `preceding_ai_context` semantics (the previous AI turn must already be in the session when this turn's user message is ingested). **peer `memoweft ^0.5 || ^0.6`** — the first adapter to span both: the assistant-reply / preceding-context line uses `recordAssistantReply` (a 0.6 method) behind a runtime capability probe, so a 0.5 host degrades to recall + user/tool ingestion with no error. **Privacy** — `provenance` / `contentType` / `id` / `score` never enter the injected system message (only `content` / `confidence` / `credStatus` do); provenance reaches the host only via `onRecall`. Reuses the §16.2 degradation semantics (200 ms recall timeout, ingest retry-once, injected logger; never throws to Mastra) and passes the adapter-kit contract (AD-1…AD-9; `ad5` / `ad9` N/A). All seams were pinned from `@mastra/core@1.51.0`'s type definitions, not documentation. Ships a runnable example + bilingual README (including how to coexist with Mastra's built-in semantic recall). **Purely additive** — a new package; Core's `src/index.ts` API surface is unchanged (`npm run api:check` still passes). See `DECISIONS.md` D-0038.

- **Consolidate now flags "the model produced resolutions but not one of them landed" (`DECISIONS.md` D-0036)** — the hardest part of diagnosing the id-truncation bug above was that **this class of failure left no signal at all**: valid JSON, no `jsonRepair` warning, a normal `llmCalls`, thousands of completion tokens — and empty writes. Consolidate now emits a `console.warn` when the model returned N resolutions and **none** of them could be matched to a real evidence, including **the first few ids the model actually wrote** — which is precisely the key that reveals a broken id contract. Deliberately narrow: **"the model produced nothing" does not warn** (that is a model/material question, and it is also the common shape of simplified test stubs), and **"some didn't land" does not warn** (3d narrowing, empty `resolvedContent`, and first-wins dedup all drop rows legitimately). **Observation only — behaviour is unchanged**: events are still marked `consolidated` exactly as before; making coverage gate the marking would need a retry counter on `event` (a schema change) and remains unimplemented pending human approval. Privacy: only ids and counts are logged, **never the utterance text** (same posture as `jsonRepair`'s default sink; an id is not user content). See `DECISIONS.md` D-0036.

- **v0.6 interaction-model upgrade — Phase 3: `formedBy`'s carrier dimension is now derived by code, not self-reported by the LLM (`DECISIONS.md` D-0035)** — a cognition's source strength along the **carrier dimension** ("whose words is this": `stated` / `confirmed` / `observed`) is now computed by a new internal `deriveFormedBy` from each supporting evidence's `sourceKind` + its semantic resolution, instead of being taken from the model's `formed_by` field. **`inferred` is deliberately still model-reported** (a scope reduction of the original "formedBy fully taken over" decision, human-approved): that field encodes a *different* dimension — how far the cognition sits from the utterance — and the risks are asymmetric (calling a `confirmed` a `stated` inflates 280→600 and is exactly what iron rule 3a guards against; saying "I inferred this" (200) under-reports and carries no incentive to lie). Deriving it from the table verbatim would have judged "user is single" — inferred from "how do I find a girlfriend" — as `stated`/600 instead of `inferred`/200, **effectively abolishing the `inferred` tier**. **Multi-evidence aggregation takes the weakest** member: "some supporting evidence is an affirmation ⇒ this cognition has at least one affirmed link" is an entailment, whereas taking the strongest assumes its non-entailing converse — with `pickSupport` checking only the id whitelist and **not relevance**, and `validEvidence` spanning a whole batch (12 conversation turns in production), taking the strongest would let an affirmation-derived cognition reach `stated`/640/`limited` merely by *also citing one unrelated volunteered utterance* — no malicious model required. **The `confirmed → stated` in-place upgrade is removed**: its gate depended on the very `formed_by` field Phase 3 stops teaching (so it would have silently died, quietly reversing D-0033), and rewriting a cognition's source label in place contradicts "surface conflicts, don't adjudicate them" — a later volunteered statement now **forms a second, coexisting `stated` cognition** while the original `confirmed` stays on file (same proposition may exist twice; dedup deferred). Fallback for "spoken but no resolution" (measured: **17.5%** of spoken evidence) uses a **structural fact** rather than the table's literal `inferred` default: *no preceding AI turn ⇒ no proposition to affirm ⇒ `proposition_origin` can only be `user_stated`* → `stated`. **Full-corpus verification** (49 scenarios on live mimo, vs the v5 + Phase-2 baseline): **not one formedBy/resolution assertion failed**; short-reply held **51/51** even though formedBy went from *label-picking* to *derived-from-comprehension*; spoken resolution coverage rose **82.5% → 100%**; `overInferRate` stayed **0.00 throughout**. Overall structural assertions moved 97.1% → 96.0% — **every one of those failures is `contentType`, none is `formedBy`**: the known fact-vs-state grey zone from `DECISIONS.md` D-0019 (one-off events fit neither `fact` nor `state`), with some scenarios flipping green and others red, at the same magnitude D-0019 itself judged to be single-run variance. **Purely internal**: `deriveFormedBy` is not exported (no consumer, per iron rule 4) and the API snapshot is unchanged. See `DECISIONS.md` D-0035.
- **v0.6 interaction-model upgrade — Phase 2 (in progress): the semantic resolver produces and persists `semantic_resolution` (`DECISIONS.md` D-0034)** — `consolidate` now persists one `semantic_resolution` row per real evidence alongside the cognitions it forms: `resolvedContent` (what the user's words actually assert once the AI's preceding question is resolved — "是" → "用户确认喜欢爬山") plus four analysis dimensions (`responseAct` affirm/negate/select/elaborate/ask/none/other · `promptAct` propose/ask/state/none/other · `propositionOrigin` user_stated/assistant_proposed · `assertionStrength` explicit/weak/none) and an optional `requiredContext`. **Iron rules 3a/3d by construction** — the resolver reuses consolidate's existing `validEvidence` whitelist, so a resolution citing a **forged or AI-context evidence id is structurally dropped**, and `resolvedContent` is an *interpretation, not evidence*: it never mints an evidence id and can never enter the support whitelist (a cognition still sources only to the user's real utterance). Writes are **idempotent** (an evidence that already carries a resolution is skipped) and each row records `resolverVersion` (`consolidate@<prompt version>`) for traceability; **illegal enum values converge to `null`** instead of writing dirty data. **Phase 2 scope is produce + store only — it deliberately does not touch `formedBy`** (that is Phase 3's `deriveFormedBy`). **Verified against a live model** (it began inert — the prompt had not been taught yet): the consolidate prompt was bumped **v3 → v5**, teaching the model to (a) read the ⟨preceding AI turn⟩ suffix as *context only, never evidence*; (b) label a merely-nodded-along proposition `confirmed` rather than `stated`, with explicit branches for **denial** (the user negating the AI's guess asserts the *negated* proposition in their own voice → `stated`), **hedged nods** ("maybe, I guess" → prefer forming nothing; if formed, still `confirmed` and worded tentatively), **selection** ("the former" → `confirmed`; a human decision — the pointer carries no content of its own, so resolving it depends entirely on the AI's turn), and a **carve-out against the small-talk guard** (the guard's "haha/ok/sure" means *idle* fillers — ones with no suffix, or from which no specific proposition can be recovered); (c) apply **narrow scope** (a long AI passage + a vague "ok" forms nothing); and (d) emit `resolutions`. **Full-corpus before/after on real mimo** (49 scenarios × 2 rounds, prompt as the only variable): overall structural assertions **90.5% → 97.1%** (248→266/274), fully-green scenarios **31 → 41**, `overInferRate` **0.00 throughout**; the new short-reply discipline **36/51 → 51/51 (+29.4pp)** and **none of the six pre-existing disciplines regressed** (chitchat-negative held 35/35 — the adversarial review's top regression suspect, since the guard's "嗯" and the affirmation rule's "嗯" collide literally). Worth noting: v3 already scored gistRecall **0.90** on short-reply — the model always *understood* what "yeah" confirmed; what Phase 2 adds is the **sourcing discipline and structured output**, not comprehension. The structure, structural wall, idempotency, enum convergence, source narrowing, and the no-store boundary are additionally pinned by 10 offline stub-LLM tests. **Additive**: `ConsolidateDeps.semanticResolutionStore?` / `UpdateProfileDeps.semanticResolutionStore?` are optional — a caller that does not pass the store (old callers, bench, unit tests) persists nothing and behaves verbatim as before (pinned by a dedicated test; api snapshot refreshed via `api:update`). See `DECISIONS.md` D-0034.
- **v0.6 interaction-model upgrade — Phase 1: context infrastructure (`DECISIONS.md` D-0034)** — introduces the structured interaction-semantics foundation so MemoWeft can understand context-dependent short replies ("yes" / "the latter" / "maybe"). Two new tables (via the store's construct-time `CREATE TABLE IF NOT EXISTS`, same pattern as `management_log`, **not** formal migrations — `LATEST_SCHEMA_VERSION` stays v1, the schema-convergence test covers them): `interaction_context` (a snapshot of user-visible context per conversation/episode — **never a Cognition, never evidence**, iron rule 3a) and `semantic_resolution` (per-evidence semantic resolution — **structure only in Phase 1**; the resolver populates it in Phase 2). **Core now owns conversation context** so the bare-ingest path benefits — the headline fix: the real product weftmate goes 100% through bare `ingestUserMessage`, never `handleConversationTurn`, so D-0033's affirmation capture never actually fired there. `ingestUserMessage` gains `conversationId?` / `episodeId?`; passing `conversationId` makes core maintain a per-conversation `InteractionSession` that captures the previous AI turn into the evidence's `preceding_ai_context` (**reusing D-0033's structural-wall column, so the existing distill/consolidate injection works verbatim on the bare-ingest path — zero Cognition-logic change**) and persists one `interaction_context`. New facade method `recordAssistantReply({ conversationId, content })` lets a host with its own agent loop report the AI reply as follow-on context (**into the window only, never evidence**, 3a). Portable bundle bumped to **schemaVersion 2**: `data` gains optional `interactionContexts?` / `semanticResolutions?` (v1 bundles import as empty) and `ImportPlan.counts` gains the two counts; the structural wall is unaffected (interaction context carries AI text but is an independent table that never enters the consolidate whitelist — 3a/3d are orthogonal to whether it rides in a bundle). **Additive**: new exported types (`InteractionContext` / `SemanticResolution` / `VisibleTurn` / `ResponseAct` / `PromptAct` / `PropositionOrigin` / `AssertionStrength` / the two store interfaces / `RecordAssistantReplyInput`), optional fields (`UserMessageInput` / `ConversationInput` / `MemoryBundle.data` / `ImportPlan.counts` / `ExportDeps` / `ImportDeps` / `StoreBundle`), a new method, and `BUNDLE_SCHEMA_VERSION 1 → 2` (api snapshot refreshed via `api:update`). Phase 1 focuses on the bare-ingest path; the `handleConversationTurn` path's `interaction_context` persistence is deferred to Phase 2. See `DECISIONS.md` D-0034.
- **Affirmation / AI-context capture — structural plumbing that lets an orphan "yes" become a `confirmed` cognition (`DECISIONS.md` D-0033, Phase 1b)** — completes the D-0033 mechanism so information that lives **only in the AI's turn** ("AI: you like hiking? / user: yes") can be captured without ever letting the AI's words become evidence. The Conversation path captures the previous AI turn and persists it as a **write-only, read-back-blocked `preceding_ai_context` column** on the user's evidence row (new `EvidenceInput.precedingAiContext?`; the column is added via the store's idempotent column-fill, same pattern as `mutedAt` — old DBs upgrade transparently). It is injected as **read-only context** into distill/consolidate (behind the existing tier + `allowInference` privacy filter) so the model can see *what* the user affirmed — but it is **deliberately absent from the `Evidence` read shape and never mapped by `fromRow`**, so `listEvidence` / `exportBundle` / MCP / `TurnOutcome` all structurally cannot surface it (the AI's words never leak out as evidence — a structural guarantee, not strip-based). The only reader is the internal `EvidenceStore.precedingAiContextOf(id)` used for injection; it never mints an evidence id, so the consolidate support-id whitelist can never cite it — **iron rule 3a/3d hold by construction** (an affirmation cognition sources only to the user's real utterance). A `confirmed` cognition is lifted above its 480 cap **only** when the user later *volunteers* it: `consolidate.reinforce` upgrades `confirmed → stated` when the LLM classifies a reinforcement as `stated` **and** at least one cited supporting utterance is `spoken`-sourced (a structural backstop — `observed`/`tool`/`inferred` can never trigger the upgrade), persisted via a new `CognitionPatch.formedBy?`. An adversarial suite pins the anti-laundering invariants: an AI leading-question storm with repeated "yes" stays `confirmed`, capped ≤ 480, never `limited`/`stable`, and is excluded from trend aggregation; the AI text is never a source; and neither an `observed` reinforcement nor a bare re-affirmation triggers the upgrade. **Additive**: `EvidenceInput.precedingAiContext?`, `EvidenceStore.precedingAiContextOf`, and `CognitionPatch.formedBy?` (api snapshot refreshed via `api:update`); the `Evidence` read shape is unchanged. Phase 2 (distill/consolidate prompt teaching + eval) activates the model behavior. See `DECISIONS.md` D-0033.

## [0.5.1] — 2026-07-15

Recall quality, source-aware consolidation, deterministic time, traceable recall controls, and repository adapter workspaces. The runtime feature scope is based on committed `main` at `4e4a96a`; the release commit adds only delivery documentation, CI safety, and lint-gate fixes. The later D-0033 conversation-context work is not included.

> **Root npm scope:** the `v0.5.1` tag publishes only `memoweft@0.5.1`. Entries below that name `@memoweft/*` packages describe repository workspace integration state; those packages are not bundled into the root tarball and require their own versioned npm releases.

### Changed

- **Useful recall without an embedder (`DECISIONS.md` D-0017)** — the default no-embedder path now uses `KeywordRetriever` instead of `NullRetriever`, so a zero-configuration host can retrieve matching memories rather than always receiving an empty recall result. Supplying an embedder still selects the vector path as before.
- **Source-aware consolidation (`DECISIONS.md` D-0018)** — distillation and consolidation now carry evidence-source labels through the model prompt and map them to `formedBy`, so observed and tool evidence are no longer silently treated as user-stated memory. This hardens source-strength discipline without changing the public API or schema.
- **Default `config.profileUpdate.batchSize` 5 → 12 (`DECISIONS.md` D-0032)** — a host consolidates the profile (`updateProfile`) after this many new conversation turns (or `idleMinutes` idle, whichever first). Raised from 5 to 12 as a dogfood tuning: fewer consolidation runs means the (growing) profile is re-sent to the model fewer times → **lower token cost** (the config's own comment already noted "太勤又费"), and distill sees more of the user's surrounding turns per batch → **better reference resolution**. The idle fallback (30 min) is unchanged. Default-value only — the config **type** is unchanged, so the API-surface snapshot is unaffected (`api:check` still passes); hosts read the value dynamically and follow automatically.

### Added

- **New `FormedBy` value `confirmed` — source-strength layer (`DECISIONS.md` D-0033, Phase 1a)** — a 5th source-strength between `inferred` and `observed` for a cognition the user formed by **affirming an AI-proposed guess** (AI: "你喜欢爬山吧?" → user: "是的"), rather than volunteering it. Base score 280 with a **structural cap**: even fully reinforced it maxes at 480 (< `limited` 500), so a purely-affirmed cognition can never reach `limited`/`stable` — only the user later *volunteering* it (a `stated` utterance) lifts it above the cap. `confirmed`-sourced cognitions are **excluded from trend aggregation** (a structural guard so an AI's leading-question storm cannot launder into a confident `ruled` trend). A regression test pins the cap invariant. **Additive** to the exported `FormedBy` enum (hosts must keep a `default` branch, per the contract's break policy); the `config.consolidation.baseByFormedBy` type gains a `confirmed` key (api snapshot refreshed via `api:update`). This lands the source-strength scaffolding only — the AI-context capture that *produces* `confirmed` cognitions is staged for later phases (see `DECISIONS.md` D-0033).
- **New adapter — `@memoweft/adapter-llamaindex` for LlamaIndex (`DECISIONS.md` D-0029)** — a memory-block + stream-tap adapter that wires MemoWeft's long-term memory into LlamaIndex's agent framework. `createMemoWeftLlamaIndex(core, opts)` returns `{ memoryBlock, persistFromAgentStream, persistUserTurn, formatKnowledge }`. **Recall** goes through a `MemoWeftMemoryBlock` (extends `BaseMemoryBlock`): drop it into `createMemory({ memoryBlocks: [block] })` and its `get(messages)` recalls memory and returns it as a neutral `role:'memory'` message injected each turn; `put()` is a deliberate no-op (Memory hands `put` the whole conversation incl. assistant output / already-injected context — ingesting there would store dirty evidence). **Writes** go through `persistFromAgentStream(core, agent.runStream(userMsg), { userMessage, originId })` — a pass-through async generator that re-yields every event unchanged and, along the way, stores the user's turn (`ingestUserMessage`) and each tool **result** (`ingestToolResult`). **Iron rule 3a by construction** — it ingests only events matched by `agentToolCallResultEvent.include(...)` (the tool's actual return); `agentToolCallEvent` (the LLM's call intent) is a different event type the result matcher never accepts. **Privacy** — `provenance` never enters the injected memory message; it reaches the host only via `onRecall`. Reuses the §16.2 degradation semantics (never throws to / interrupts the stream) and passes the adapter-kit contract (AD-1…AD-9; `ad5` / `ad9` N/A). **Upstream caveat**: LlamaIndex.TS's granular `@llamaindex/*` packages are marked deprecated while its umbrella `llamaindex@^0.12` (which this adapter peers on) still depends on them; the direct dependency on the deprecated `@llamaindex/core` was eliminated (the umbrella re-exports it), but `@llamaindex/workflow` — the event-driven agent API's only home, itself a dependency of the umbrella — remains a direct peer (see `DECISIONS.md` D-0029). **Purely additive** — a new package; Core's `src/index.ts` API surface is unchanged.
- **New adapter — `@memoweft/adapter-langchain` for LangChain (`DECISIONS.md` D-0028)** — a retriever + callback adapter that wires MemoWeft's long-term memory into `@langchain/core`. `createMemoWeftLangChain(core, opts)` returns `{ retriever, writeCallback, formatKnowledge, persistUserTurn }`. **Recall goes through a `MemoWeftRetriever` (extends `BaseRetriever`)**, not a callback — LangChain callbacks are observe-only (they can't inject), so `_getRelevantDocuments(query)` calls `core.recall` and returns `Document[]` (`pageContent` = the cognition's content; `metadata` = confidence / credStatus / id / contentType / score); `formatKnowledge` renders them into a neutral knowledge block for your prompt. **Writes**: `MemoWeftWriteCallback` (extends `BaseCallbackHandler`) implements **only `handleToolEnd`** — the tool's actual result → `ingestToolResult` — and **never declares `handleToolStart`**, so the LLM's call intent is physically undeliverable to it (iron rule 3a by construction); `persistUserTurn({ text, originId })` stores the user's own words. **Privacy** — `provenance` never enters a `Document` (not in `pageContent`, not in `metadata`); it reaches the host only via `onRecall`. Reuses the §16.2 degradation semantics and passes the adapter-kit contract (AD-1…AD-9; `ad5` / `ad9` N/A). Deliberately does **not** implement `BaseChatMessageHistory` / `BaseMemory` (those model short-term turn history, orthogonal to MemoWeft's profile-level memory). Ships a runnable example + bilingual README. **Purely additive** — a new package; Core's `src/index.ts` API surface is unchanged. See `DECISIONS.md` D-0028.
- **New adapter — `@memoweft/adapter-openai-agents` for the OpenAI Agents SDK (`DECISIONS.md` D-0027)** — a run-wrapper adapter that wires MemoWeft's long-term memory into `@openai/agents`. `createMemoWeftRunner(core, opts)` returns `{ run, callModelInputFilter, persistToolOutputs }`: `run(agent, input, options)` captures the user's turn (`ingestUserMessage`, spoken), chains a `callModelInputFilter` that injects recalled memory into the model's **`instructions`** (so the captured user input stays injection-free by construction), and after the run scans `RunResult.newItems` to persist tool results (`ingestToolResult`). **Iron rule 3a by construction** — it only reads items of `type === 'tool_call_output_item'` (the tool's actual return) and their `output` / `callId`, never `tool_call_item` (the LLM's call intent). **Privacy** — `provenance` / `contentType` / `score` never enter the injected `instructions` (only `content` / `confidence` / `credStatus` do); provenance reaches the host only via `onRecall`. A host that already passes its own `callModelInputFilter` is chained (host runs first, recall appended), and `@openai/agents` is dynamically imported so the filter / ingest paths can be tested without loading the SDK. Reuses the §16.2 degradation semantics (200 ms recall timeout, ingest retry-once, injected logger; never throws to the SDK) and passes the adapter-kit contract (AD-1…AD-9; `ad5` / `ad9` N/A). Ships a runnable example + bilingual README. **Purely additive** — a new package; Core's `src/index.ts` API surface is unchanged (`npm run api:check` still passes). See `DECISIONS.md` D-0027.
- **New adapter — `@memoweft/adapter-claude-agent-sdk` for the Claude Agent SDK (`DECISIONS.md` D-0025)** — a hooks-based in-process adapter that wires MemoWeft's long-term memory into `@anthropic-ai/claude-agent-sdk`. `createMemoWeftAgentHooks(core, opts)` returns `{ hooks }` you spread into `query({ options: { hooks } })`: a **`UserPromptSubmit`** hook injects recalled memory (as `additionalContext`) and persists the user's turn (`ingestUserMessage`, spoken), and a **`PostToolUse`** hook persists tool results (`ingestToolResult`). **Iron rule 3a by construction** — the tool hook reads only `tool_response` / `tool_use_id` and **never** `tool_input` (the LLM's call intent), and `PreToolUse` is never mounted, so call intent cannot become evidence. **Privacy** — `provenance` / `contentType` / `score` never enter the injected `additionalContext` (it uses only `content` / `confidence` / `credStatus`); provenance reaches the host only via `onRecall`. Reuses the §16.2 degradation semantics (200 ms recall timeout, ingest retry-once, injected logger; **never throws to the SDK**). Passes the shared adapter-kit contract (AD-1…AD-9; `ad5`/`ad9` declared N/A like the read middleware). Also documents the near-zero-code alternative — mounting `@memoweft/mcp-server` via the Agent SDK's `mcpServers` option — and ships a runnable example + bilingual README. **Purely additive** — a new package; Core's `src/index.ts` API surface is unchanged (`npm run api:check` still passes). See `DECISIONS.md` D-0025.
- **Recall v2 end-to-end — both adapters pass the new recall surface through, plus a new MCP mute tool (`DECISIONS.md` D-0024)** — the D-0021/22/23 recall-quality surface (`explain`→`provenance`, the `contentTypes` filter + `contentType` on results, mute negative feedback) now reaches hosts **through the two official adapters**. **`@memoweft/adapter-ai-sdk`**: `MemoWeftMiddlewareOptions` gains `contentTypes?` / `explain?` (passed straight to `core.recall`), and the `onRecall(items)` callback now carries each item's `id` / `contentType` / `score` and — when `explain: true` — its `provenance` chain (each brief with `allowCloudRead` / `allowInference`). **Privacy hard constraint: `provenance` never enters the injected prompt** (`buildKnowledgeBlock` still uses only `content` / `confidence` / `credStatus`) — evidence verbatim + authorization bits would otherwise bypass tier filtering; provenance reaches the host only via `onRecall`, which the host tier-filters before forwarding to a cloud model. **`@memoweft/mcp-server`**: `memoweft_recall` gains `contentTypes?` / `explain?` inputs and returns `contentType` per item plus (with `explain`) a `provenance` chain that is **tier pre-filtered in the tool layer** — cloud-readable evidence keeps its `summary`, cloud-restricted evidence has its summary withheld and returns only `{ evidenceId, relation, sourceKind }` + the authorization bits (so the host can still self-filter). A **new light write tool `memoweft_mute_cognition`** (wraps `core.memory.muteCognition`) takes `WRITE_TOOL_NAMES` **2 → 3** and `ALL_TOOL_NAMES` **7 → 8**: it toggles whether a cognition is excluded from recall (recall negative feedback), the cognition stays active and keeps evolving the profile, it is orthogonal to confidence (iron rule 3b), and it does not delete or change any cloud-read authorization — so `muteCognition` moves off the MCP server's "never registered" blacklist into the whitelist. **Additive** for the ai-sdk options and the recall inputs; the mute tool is a whitelist expansion (a reversible, authorization-neutral, confidence-orthogonal light write — existing clients that don't call it are unaffected). Core's `src/index.ts` API surface is unchanged (`npm run api:check` still passes — this all lives in the adapter packages). See `DECISIONS.md` D-0024.
- **Recall negative feedback (mute) — `core.memory.muteCognition(...)` (`DECISIONS.md` D-0023)** — a host can act on "this recalled memory wasn't useful" by muting a cognition: `muteCognition({ cognitionId, muted: true, reason })` sets a new `mutedAt` timestamp so **recall skips it, while it stays active and keeps evolving the profile** — a lighter signal than `archiveCognition` (which shelves it from consolidation too) or `invalidateCognition` (which marks it no-longer-true). `muted: false` un-mutes. Mute is **orthogonal to confidence** (iron rule 3b — `muteCognition` never touches `confidence`/`credStatus`). New `Cognition.mutedAt?`, `MuteCognitionInput`, and the `muted_at` column (added via the store's idempotent column-fill, same as `archivedAt`; old DBs upgrade transparently, and it rides along in portable bundles). **Purely additive**: a nullable column, an optional method, and an optional cognition field. See `DECISIONS.md` D-0023.
- **Recall content-type filter — `recall({ contentTypes })` + recalled cognitions expose their `contentType` (`DECISIONS.md` D-0022)** — `RecallInput` gains an optional `contentTypes?: ContentType[]` allow-list; when set, `recall` returns only cognitions of those types (e.g., `['fact', 'preference']` to skip transient `state`). Every recalled item now also carries its `contentType` (`RecalledCognitionItem.contentType` required; `RecalledCognition.contentType?` optional) so hosts can see each memory's type. The filter is applied in the `core.recall` facade (the low-level `recallCognitions` only populates the field); it is a **post-filter of the top-K**, so — like the existing similarity/decay gates — it can under-fill (return fewer than `topK`, or empty, if the top-K hold no matching type). **Purely additive**: `contentTypes` absent = all types (unchanged). See `DECISIONS.md` D-0022.
- **Recall explanation — `recall({ explain: true })` returns each cognition's supporting evidence chain (`DECISIONS.md` D-0021)** — `RecallInput` gains an optional `explain?: boolean`; when `true`, every recalled `RecalledCognition` carries a new optional `provenance?: RecalledEvidence[]` listing the evidence it's built on (`RecalledEvidence = { evidenceId; relation; summary; sourceKind; allowCloudRead; allowInference }` — the evidence brief **carries authorization flags** so a host can tier-filter provenance before forwarding it to a cloud model, aligned with `buildMemoryGraph`). This surfaces MemoWeft's core "traceable memory" value — a host can show *why* a memory surfaced (what evidence backs it), not just a bare similarity `score`. The enrichment happens in the `core.recall` facade (which already holds both stores), so the low-level `recallCognitions` / `RecallDeps` are unchanged. **Purely additive**: `explain` defaults off (no extra queries, identical behavior), `provenance` is optional. Host-facing only — it does not weaken the cloud-read privacy model (which governs what write-path cloud models read via tier filtering) and never enters confidence self-computation. This first increment covers the **evidence chain** (already in the store); "which query terms matched" is deferred (that information is lost at the retriever's `{id, score}` layer). See `DECISIONS.md` D-0021.
- **Injectable clock (`Clock` / `systemClock`, Phase 4)** — `createMemoWeftCore({ clock })` and `openStores(dbPath, cfg, clock)` accept an optional `clock: Clock` (`() => Date`) that is the time source for store writes (`recordedAt` / `created_at` / `updated_at`). Injecting a fixed or advancing clock gives **determinism** (two runs produce identical timestamps) and **time-travel** (advance the clock to age out transient state while facts persist) — the foundation for the Phase 4 deterministic, no-key demo. Defaults to real system time, so existing callers are unaffected (**purely additive**). The clock only produces timestamps and **never enters confidence self-computation** (iron rule 3b: decay is still computed read-time from `updatedAt` vs `clock()`). Beyond the stores, the injected clock flows through the **entire facade path** (consolidate / attribute / management-audit / read-path decay `now`) via `createMemoWeftCore({ clock })`. The two **non-facade** paths — proactive asking (`ProposeAskDeps` / `RevisitDeps`, `askedAt`) and the dev run-log (`RunLoggerOptions`, `ts`) — take their own optional `clock?`, completing "every time source is injectable" (no stray `new Date()` timestamps left). All additive. See `DECISIONS.md` D-0015 (facade) and D-0020 (asking / run-log tail).
- **Tool-result ingestion — `SourceKind` gains `'tool'` + `core.ingestToolResult` (AD-3, memory surface contract §16.1; `DECISIONS.md` D-0013)** — MemoWeft can now record a tool execution's **returned result** (an external, objective data point = legitimate evidence) as `tool` evidence. New Core facade method `ingestToolResult(input: ToolResultInput): Promise<Evidence>` stores a verbatim result payload as one idempotent `tool` evidence. **Privacy:** tool evidence defaults to local-only (`allowCloudRead=false`) through `config.toolDefaults`, enforced again in `evidenceStore.put()`. **Iron rule 3a boundary:** only the returned output is ingested, never the model's tool-call intent or arguments. The memory graph adds `toolEvidenceCount` and a distinct tool node color; repository adapter workspaces add matching ingestion surfaces. This is additive: `source_kind` is free text and the new facade/config/graph types require no schema migration. The earlier source-label propagation gap is resolved in the same release by D-0018: distillation and consolidation now preserve source labels when determining `formedBy`, so tool or observed evidence is not silently treated as user-stated. See the adapter-kit AD-3 contract and `DECISIONS.md` D-0013/D-0018.
- **Adapter degradation semantics (memory surface contract §16.2)** — the two official adapters (`@memoweft/adapter-ai-sdk`, `@memoweft/mcp-server`) now **degrade instead of interrupting the conversation** when the memory layer fails or times out. The read path (`recall`) is wrapped in a **200ms timeout, configurable** via a new optional factory option `recallTimeoutMs` (`createMemoWeftMiddleware(core, { recallTimeoutMs })` / `createMcpServer(core, { recallTimeoutMs })`); a timeout counts as a failure. The **read path does not retry** — it degrades immediately (injects an empty context / returns an empty tool result with `isError: false`); the **write path (ingest) retries once** before giving up. Both adapters take a new optional `logger` that records **structured degradation events only** (shape `{ event: 'memory_degraded', op: 'recall' | 'ingest', reason: 'timeout' | 'error' }`, aligned across both adapters; the MCP server adds an optional `tool` field) — never user content, verbatim text, or secrets; no logger = silent (the previous behavior). This turns the MCP server from "a memory-layer throw surfaces as a protocol error / crashes the process" into "degrade and keep the conversation going", while **invalid parameters and other caller errors are still surfaced as real errors, not swallowed as degradation**. Purely additive hardening: the new factory options are optional (unset = previous behavior plus the built-in 200ms timeout and silent degradation), and **Core's public API surface is unchanged** (`npm run api:check` still passes — the timeout/retry/logger live entirely inside the adapter packages). New exported types for host-typed loggers: `MemoWeftLogger` / `MemoWeftDegradedEvent` (ai-sdk) and `McpServerLogger` / `McpDegradedEvent` (mcp-server). See the memory surface contract §16.2 and `DECISIONS.md` D-0012.

## [0.5.0] — 2026-07-06

Ecosystem release: an MCP server, a Vercel AI SDK adapter, and LLM token usage accounting — plus the local/cloud model tiers and plugin-contract v2 that landed on `main` since 0.4.0. Core stays zero runtime dependencies.

### Added

- **Local/cloud model tier for the write path (`ModelTier = 'cloud' | 'local'`)** — the write-path privacy gate is now `filterReadableByTier(items, tier)` instead of a fixed cloud filter. Set `MEMOWEFT_WRITE_LLM_TIER=local` (default `cloud`) so a **local write model digests `observed` behavioural evidence** (cloud=false by default) into the profile — the behavioural-observation collection path is now a real closed loop, not "authorize upload or it never gets digested." The tier is a new optional field on `LLMConfig` / `LLMClient` (`ModelTier` is exported; unset = cloud, non-breaking for hosts that inject their own `LLMClient`), bound to the client instance so a missing write model that falls back to the chat model inherits the chat tier (no "declared local, actually cloud" leak). The config wizard (`/api/gen-env`) emits `MEMOWEFT_WRITE_LLM_TIER` and warns when no local write model is configured.
- **`DistillResult.tierBlockedCount`** — how many pending evidence the current write-model tier cannot read (surfaced via `updateProfile().distilled`), so a host can tell when `observed` evidence is waiting for a local model or cloud authorization.
- **Plugin contract v2 (`MemoWeftPlugin` + hooks + `PluginContext`)** — the plugin contract moves into Core (`src/plugin/`, exported, experimental) and grows real hooks: `onLoad` / `onUserMessage` / `onObservation`, fired at the Core method layer (the conversation and ingestion pipelines are untouched). Register via `createMemoWeftCore({ plugins: [...] })` (optional — unset behaves as before). Hooks are **observe-only**: return values are discarded, they cannot mutate the reply/message, and a throwing hook is logged without crashing the turn or ingestion. Each hook gets a restricted `PluginContext` (`submitObservation` + `requestMemory`) that is closure-built (never hands over the stores), permission-gated by a declarative `permissions` field, and cannot mark a submitted observation cloud-readable (auth bits are stripped → conservative `observed` defaults). The reference host lists registered plugins and their declared permissions in a new "Plugins" tab (`GET /api/plugins`). See `docs/plugin-contract.md` and the runnable `examples/plugin-hook.ts`.
- **Collector platform seam (`createForegroundSampler`)** — `@memoweft/collector-active-window` now selects its foreground-window sampler by platform through a factory; the collection loop is platform-agnostic. Adding macOS/Linux means adding one sampler case (node built-ins only, no npm deps). Windows is the only implemented platform today; other platforms exit with an explicit "how to add it" message.
- **LLM token usage accounting (`core.usage()`)** — the built-in `OpenAICompatClient` / `OpenAICompatEmbedder` now capture the `usage` returned by OpenAI-compatible endpoints (previously read off the wire and discarded) into a monotonic per-instance `UsageStats` (`promptTokens` / `completionTokens` / `totalTokens` / `callsWithUsage`), and `core.usage()` reports the running total bucketed into `llm` / `embed` / `total` so a host can price its own spend. **Raw counts only — the library ships no price table** (pricing drifts by vendor/time and is the host's assumption; multiply the counts yourself). Endpoints that don't return `usage` (common for local models) are skipped silently: the count stays 0, nothing crashes — hence `callsWithUsage` (≤ `callCount`) so a host can compute an honest average. `usage` is a new **optional** field on `LLMClient` / `Embedder` (unset for hosts that inject their own — non-breaking, same as `tier?`); it never flows into self-computed confidence. The reference host exposes `GET /api/usage`.
- **`@memoweft/mcp-server` (new package)** — a Model Context Protocol server that exposes MemoWeft's memory to external AI clients as MCP tools. **Six tools only**: five read (`memoweft_recall`, `memoweft_list_cognitions` / `_evidence` / `_events`, `memoweft_graph`) and one light write (`memoweft_ingest_user_message`, which stores a single verbatim user message as spoken evidence — no profile update, no digestion, no authorization grant). Destructive operations, authorization changes, and full-conversation digestion are **intentionally not registered as tools**, so an external LLM cannot delete memory, flip cloud-read bits, or rewrite the profile. Ships as a standalone publishable package — `@modelcontextprotocol/sdk` lives in *its* dependencies; Core's stay `{}`. Registry submission is a manual publishing step.
- **`@memoweft/adapter-ai-sdk` (new package)** — a Vercel AI SDK adapter. **Read**: `createMemoWeftMiddleware(core)` wraps a language model via `wrapLanguageModel` and injects recalled memory into the prompt through `transformParams`, reusing Core's neutral knowledge-block wording (low-confidence items flagged as guesses). **Write**: `createPersistOnEnd(core, { userMessage })` persists the user's verbatim turn on `onEnd` — the assistant reply is never stored, and because the user message is captured explicitly (not scraped from the result event) the injected recall can't leak back in as "user input." `ai` is a peer dependency; Core's dependencies stay `{}`.

### Changed

- **Coverage fix:** `distill` now marks as covered only the evidence it actually digested into the event. Previously, when `observed` (cloud=false) evidence shared a batch with cloud-readable evidence, it was marked covered without ever being digested and could not be recovered — switching to a local model or authorizing upload later would not re-process it. Blocked evidence now stays pending and re-scannable.
- **`allowInference` gate is now enforced in `distill` and `consolidate`** (previously only in `attribute`), keeping all three write-path steps consistent. Consequence: evidence with `allowInference=false` **and** `allowCloudRead=true` no longer feeds the profile on the cloud path either (it used to). This state does not exist by default — it requires explicitly revoking inference on a specific piece of evidence — so most deployments see no change; where it applies, it is the authorization bit taking effect as documented.

## [0.4.0] — 2026-07-05

English-first defaults with a zero-dependency bilingual layer, model-compatibility knobs (configurable temperature + reasoning-model response handling), and the Memory Surface Contract v1.

### Added

- **Bilingual layer (`config.language`)** — every prompt sent to the model and every host/user-facing message (errors, fallbacks, template questions) now resolves to English or Chinese through a zero-dependency constant table. Set `config.language` at runtime, or `MEMOWEFT_LANG=zh`; a new `type Lang` is exported. The cognitive-discipline instructions were translated with equivalent meaning (conflict exposure, no self-corroboration, hypothesis capping), with the offline eval suite kept green as the guard.
- **Configurable generation temperature** — `LLMConfig.temperature`, read from `MEMOWEFT_LLM_TEMPERATURE` / `MEMOWEFT_WRITE_LLM_TEMPERATURE` so chat and write paths can differ. Unset keeps the previous default of `0.3` (no behaviour change); `0` is honoured. Temperature only enters the generation request — it never flows into MemoWeft's self-computed confidence.
- **Reasoning-model response compatibility** — the client strips paired `<think>…</think>` segments from responses (an unclosed tag is left intact so a real answer is never dropped), and `extractJsonObject` now uses a brace-balanced scan instead of a greedy `lastIndexOf`, so a reasoning model's thinking (with stray braces) no longer poisons write-path JSON parsing.
- **Memory Surface Contract v1** — the public surface (`import 'memoweft'`) now carries a documented stability contract instead of unlabelled exports. New [`docs/memory-surface-contract.md`](docs/memory-surface-contract.md) is the single source of truth for hosts: it lists every facade method and data shape with a `stable` / `experimental` / `internal` tier, the breaking-change policy, and the implicit contracts hosts most often trip on. Each export group in `src/index.ts` carries a matching `// [stable]` / `// [experimental]` / `// [internal]` line (grep-verifiable).
- **Two more examples** — `examples/memory-management.ts` (controlled `core.memory.*`) and `examples/portable-bundle.ts` (portable-bundle round-trip, runs without a model). Examples now import by package name (`from 'memoweft'`; build first).
- **English documentation** — `docs/INSTALL.md` and `docs/integration.md` are now English (with `.zh-CN.md` counterparts and cross-links), matching the English-first README. A new note records that data at rest is unencrypted — disk encryption is the host/OS responsibility.

### Changed

- **Default output language is now English (`en`).** Chinese hosts that relied on the implicit Chinese output will see event summaries / hypotheses / questions produced in English (the cognitive structure is unchanged). Set `config.language = 'zh'` or `MEMOWEFT_LANG=zh` to restore Chinese.
- **Default `hostId` changed from `'testbench'` to `'local'`.** This affects only the `host_id` stamped on newly-written evidence (it is not a query key); existing databases read unchanged. The `subjectId` default (`'owner'`) is unchanged.

## [0.3.0] — 2026-07-05

补漏加固批次：隐私红线 B 下沉 Core、`prepublishOnly` 发布保险丝、加固版 JSON 解析、SQLite `busy_timeout`、可选 `better-sqlite3` 驱动（支持 Node 20/22）、扫尾、以及「永远全开源」公开承诺。

### Added

- **Optional `better-sqlite3` driver — adds Node 20 / 22 support (Node ≥24 stays the tested, zero-dependency default).** The built-in `node:sqlite` module only stabilized in Node 24, which shut out the still-large Node 20/22 install base. There is now a second SQLite driver behind the same internal seam: on Node ≥24 the built-in `node:sqlite` is used by default (still **zero runtime dependencies**), and if it is unavailable MemoWeft falls back to `better-sqlite3` when installed. Node 20/22 users run `npm i better-sqlite3` to opt in. `better-sqlite3` is declared as an **optional peer dependency** (`peerDependenciesMeta.optional`), so `npm install memoweft` still pulls **no** runtime dependencies and no native module by default. When neither driver is available, `import 'memoweft'` fails with a plain-language error listing both fixes (upgrade Node to ≥24, or install `better-sqlite3`). It is a native module usually installed as a prebuilt binary; if no prebuilt matches your platform it falls back to a `node-gyp` compile, so installation is not guaranteed on every environment.

### Changed

- **`engines.node` relaxed from `>=24` to `>=20`** — reflecting the new optional `better-sqlite3` driver. Node ≥24 remains the zero-dependency, out-of-the-box path; Node 20/22 need the optional driver installed.
- **SQLite `busy_timeout` now set to 5000ms on every self-opened connection** — when two processes write to the same database file (e.g. a host and a testbench pointed at the same `dla.db`), the later writer previously failed *immediately* with `SQLITE_BUSY`; it now waits up to 5 seconds for the write lock before erroring. Single-process use is unaffected (the synchronous SQLite API already serializes within a process). This is an infrastructure default, not a config knob. (WAL is intentionally *not* enabled in this change — it needs a matching backup-strategy change first.)
- **Observed evidence never defaults to cloud-readable** — the non-cloud default for `sourceKind: 'observed'` evidence is now enforced in `SqliteEvidenceStore.put` itself (not just the observation ingest entry). Callers that write observed evidence directly through `put` without an explicit authorization bit now get `allowCloudRead: false` by default (previously it followed the general default and could be cloud-readable). To send observed evidence to the cloud you must now pass `allowCloudRead: true` explicitly. `spoken` / `inferred` defaults and any explicitly-passed bit are unchanged; imports via `insert` are unaffected (they preserve the bundled bits).

## [0.2.0] — 2026-07

Durable schema: your `0.1.0` database upgrades losslessly on first open, with an automatic backup taken before any schema-changing migration.

### Added

- **Schema versioning & migrations** — the database now carries a `PRAGMA user_version`; `openStores` runs an ordered migration runner on open. New databases are stamped to the latest version; existing ones (e.g. a `0.1.0` database) are migrated forward, each migration in its own transaction (rolls back on failure), with an automatic pre-migration backup for schema-changing migrations and a dry-run mode. Exposed as `runMigrations` / `getSchemaVersion` / `LATEST_SCHEMA_VERSION`.
- **Downgrade protection** — opening a database whose schema version is *newer* than the running `memoweft` is now refused with a clear error, instead of silently reading/writing an unknown schema (which could corrupt data).
- A `0.1.0` database opens losslessly under the new version — verified against a frozen `0.1.0` fixture (`tests/fixtures/memoweft-0.1.0.db`) plus a schema-parity check that a freshly-created database and a migrated old one converge to the identical schema.
