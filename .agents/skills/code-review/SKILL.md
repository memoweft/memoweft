# SKILL · code-review（自查，写完即查）

**何时用**：实现一块后、跑测试前，自己先过一遍。

**目的**：在交给用户 dogfood 前，挡掉 DLA 特有的设计性错误和明显 bug。

## 通用检查

- 实现是否**只覆盖了确认过的方案**，有没有偷偷扩张？
- 可读性：命名 / 结构是否贴合周围代码？
- 错误处理、边界（空输入、空召回、首条记忆等）。

## DLA 特有坑（重点，逐条对照地图 cell 8 / cell 6）

- [ ] **系统自证**：有没有哪条路径把 DLA 自己的输出、或用户沉默 / 点赞，当成了"用户证据"喂回去？（必须没有）
- [ ] **自动消解冲突**：发现新旧矛盾时，代码是不是擅自判了一方失效？应是"标冲突 + 暴露"，不是自动选边。
- [ ] **记 ≠ 改画像**：写证据的路径有没有顺手就改了画像 / 下了结论？两者必须分开、画像更新走门槛。
- [ ] **来源强度**：亲口 / 推断 / 观测有没有被一视同仁？
- [ ] **置信度**：是不是直接把 LLM 自报的把握度当最终值了？（不允许）
- [ ] **废弃物复活**：有没有把已废的东西又写回来了——`topic` 当相关性机制、单一权重公式、State/Profile 双层时效、"不上向量"？这些都已废（见地图 cell 8/13）。
- [ ] **可迁移 / 边界**：有没有把语气 / 角色 / 表达逻辑写进 DLA？那是宿主的事（cell 9）。
- [ ] **依赖**：有没有偷偷引入新依赖或重框架？（cell 11）

发现问题 → 修；若问题源于方案本身或要改决策 → 回 task-planning 问人。
