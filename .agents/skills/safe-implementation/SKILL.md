# SKILL · safe-implementation（按方案小步写）

**何时用**：方案经人确认后，写代码时。

**目的**：把确认过的方案落成代码，且不踩 DLA 的设计红线、不擅自扩张。

## 纪律

1. **只写确认过的方案**：方案外的"顺手优化 / 未来可能用到的字段 / 健壮性加料"一律不加。要加 → 回 task-planning 问。
2. **小步**：一次一个可验证的小块，写完能在测试台看到效果。不一次铺一大片。
3. **守认知层规则（地图 cell 8）**，尤其写到相关逻辑时：
   - **记下来 ≠ 立刻改画像**：证据照记，但更新画像要走门槛。
   - **来源强度分层**：亲口 > 推断 > 观测，代码里不能一视同仁。
   - **禁止系统自证**：不得把 DLA 自身输出 / 用户沉默写成"用户证据"。
   - **冲突先暴露，不自动消解**：发现矛盾标"冲突中"，不自动判旧的失效（别学 Graphiti 那套自动作废）。
   - **不宣称证明为真**：模型自报置信度不直接当最终置信度。
4. **数据结构（cell 6）**：动 Event / 认知 / 溯源表的字段前 → 必停问人。
5. **依赖纪律（cell 11）**：TypeScript + `node:sqlite`。**能参考 / 借鉴就用，不盲目造轮子**——自由参考成熟逻辑（Mem0 / Graphiti），可用小而专、可替换的库；但**拒绝重框架（LangChain）/ 重基座（Mem0 作地基）**，核心认知层与证据真相自有。新增依赖要说清"省了什么、会不会锁定、能不能替换"。
6. **部件可替换**：嵌入器（云端优先）、LLM 客户端等做成可插拔的一处，换实现只动一个地方。

## 测试台同步

写的功能要能在测试台透视区看到对应 log；新增的内幕字段记得一并写进落盘日志（见 regression-check）。
